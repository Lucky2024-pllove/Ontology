## 规则建模概述

### 规则的定义
规则是对业务逻辑的形式化描述，定义了在特定条件下必须满足的条件或必须执行的动作。规则建模关注"业务必须遵守什么"和"在什么情况下做什么"。

### 规则的核心要素

1. **规则名称**
   - 定义：规则的唯一标识
   - 规范：描述性名称，如"付款比例校验规则"、"库存充足性检查"
   - 特征：能够反映规则的业务含义

2. **规则类型**
   - 定义：规则所属的类别
   - 类型：校验规则、业务规则、状态规则、计算规则

3. **触发条件**
   - 定义：规则何时被触发评估
   - 形式：事件触发、状态触发、时间触发
   - 示例："当订单提交时"、"当库存低于阈值时"

4. **规则条件**
   - 定义：规则评估的前提条件
   - 形式：布尔表达式、逻辑组合
   - 示例："订单金额 > 0"、"客户状态 = '正常'"

5. **规则动作**
   - 定义：条件满足时执行的操作
   - 类型：验证、计算、状态变更、通知
   - 示例："验证通过"、"计算折扣"、"发送提醒"

6. **优先级**
   - 定义：规则的执行顺序
   - 范围：数值越大优先级越高
   - 默认：0（普通优先级）

## 规则类型分类

### 1. 校验规则（Validation Rules）

#### 定义
用于验证数据的合法性、完整性和正确性。

#### 常见类型
| 校验类型 | 描述 | 示例 |
|---------|------|------|
| **必填校验** | 字段必须有值 | 订单金额不能为空 |
| **格式校验** | 值必须符合格式 | 邮箱必须符合邮箱格式 |
| **范围校验** | 值必须在范围内 | 年龄在0-150之间 |
| **唯一性校验** | 值必须唯一 | 用户名不能重复 |
| **关联校验** | 关联数据必须存在 | 客户ID必须对应存在的客户 |
| **一致性校验** | 多字段间必须一致 | 开始日期 < 结束日期 |

#### 规则示例
```
规则名称: 订单金额非空校验
类型: 校验规则
触发条件: 订单提交时
规则条件: 订单金额 != null AND 订单金额 != 0
规则动作: 
  - 如果满足: 继续后续流程
  - 如果不满足: 抛出"订单金额不能为空"异常
```

### 2. 业务规则（Business Rules）

#### 定义
定义业务运作的逻辑和约束，体现业务策略。

#### 常见类型
| 规则类型 | 描述 | 示例 |
|---------|------|------|
| **资格规则** | 定义参与资格 | 会员才能参加促销活动 |
| **限制规则** | 定义业务限制 | 单笔订单金额不超过10万 |
| **派生规则** | 计算派生值 | 订单总金额 = 商品金额 + 运费 - 优惠 |
| **分配规则** | 定义资源分配 | 订单分配给最近的仓库 |
| **路由规则** | 定义流程路由 | 金额>1万需要经理审批 |

#### 规则示例
```
规则名称: 大额订单审批规则
类型: 业务规则
触发条件: 订单提交时
规则条件: 订单金额 >= 10000
规则动作:
  - 如果满足: 订单状态设为"待审批"，发送审批通知给经理
  - 如果不满足: 订单状态设为"待支付"
```

### 3. 状态规则（State Rules）

#### 定义
定义对象状态之间的转换规则和约束。

#### 常见类型
| 规则类型 | 描述 | 示例 |
|---------|------|------|
| **状态转换规则** | 定义允许的状态转换 | 待支付 → 已支付，禁止：待支付 → 已完成 |
| **状态前置规则** | 定义进入状态的前置条件 | 只有已支付才能发货 |
| **状态后置规则** | 定义离开状态的后置动作 | 订单完成后发送评价邀请 |
| **状态超时规则** | 定义状态的超时处理 | 待支付状态24小时后自动取消 |

#### 状态转换表示例
```
当前状态 | 目标状态 | 允许? | 规则名称
---------|---------|-------|----------
待支付   | 已支付   | 是    | 支付转换
待支付   | 已取消   | 是    | 取消转换
待支付   | 已完成   | 否    | 禁止直接完成
已支付   | 已发货   | 是    | 发货转换
已发货   | 已完成   | 是    | 收货转换
```

### 4. 计算规则（Computation Rules）

#### 定义
定义派生值的计算方法和公式。

#### 常见类型
| 计算类型 | 描述 | 示例 |
|---------|------|------|
| **聚合计算** | 多值聚合 | 订单总金额 = SUM(商品金额) |
| **派生计算** | 基于其他值计算 | 折扣金额 = 原价 * 折扣率 |
| **累积计算** | 基于历史累积 | 积分余额 = 原积分 + 新增积分 |
| **分摊计算** | 按比例分摊 | 运费分摊 = 总运费 / 商品数量 |

#### 规则示例
```
规则名称: 订单金额计算
类型: 计算规则
触发条件: 订单商品变更时
规则条件: 无（始终执行）
规则动作:
  订单总金额 = SUM(商品金额) + 运费 - 优惠金额
  订单税额 = 订单总金额 * 税率
  订单实付金额 = 订单总金额 + 订单税额
```

## 规则结构定义

### 标准规则结构

```
规则名称: [规则名称]
规则ID: [唯一标识]
类型: [校验规则/业务规则/状态规则/计算规则]
描述: [规则的业务描述]

触发条件:
  - 触发事件: [事件描述]
  - 触发对象: [作用对象]
  - 触发时机: [时机描述]

规则条件:
  - 条件表达式: [布尔表达式]
  - 条件说明: [自然语言说明]

规则动作:
  条件为真时:
    - 动作1: [动作描述]
    - 动作2: [动作描述]
  条件为假时:
    - 动作1: [动作描述]
    - 动作2: [动作描述]

优先级: [数值]
生效范围: [全局/特定对象/特定场景]
版本: [版本号]
```

### 完整示例

```
规则名称: 库存充足性检查
规则ID: RULE-001
类型: 校验规则
描述: 下单时检查所选商品的库存是否充足

触发条件:
  - 触发事件: 订单提交
  - 触发对象: 订单对象
  - 触发时机: 提交前校验

规则条件:
  - 条件表达式: ∀商品项 ∈ 订单.商品列表: 商品项.数量 <= 商品项.商品.库存数量
  - 条件说明: 所有订单商品的数量都不超过对应商品的库存

规则动作:
  条件为真时:
    - 动作1: 记录"库存检查通过"日志
    - 动作2: 继续订单提交流程
  条件为假时:
    - 动作1: 识别库存不足的商品
    - 动作2: 抛出"库存不足"异常，包含具体商品信息
    - 动作3: 终止订单提交

优先级: 100
生效范围: 订单提交场景
版本: 1.0
```

## 规则优先级与冲突解决

### 优先级定义

#### 优先级数值
- **数值越大，优先级越高**
- **范围**：0-1000（建议）
- **默认值**：0

#### 优先级分类
| 优先级范围 | 分类 | 说明 |
|-----------|------|------|
| 900-1000 | 关键规则 | 必须执行，如安全规则 |
| 700-899 | 高优先级 | 重要业务规则 |
| 400-699 | 中优先级 | 一般业务规则 |
| 100-399 | 低优先级 | 辅助规则 |
| 0-99 | 默认优先级 | 普通校验规则 |

### 规则冲突类型

#### 1. 矛盾冲突
- **定义**：两条规则的条件和动作相互矛盾
- **示例**：
  ```
  规则A：如果金额>1000，则批准
  规则B：如果金额>1000，则拒绝
  ```
- **解决**：根据优先级执行高优先级规则，或人工介入

#### 2. 冗余冲突
- **定义**：多条规则执行相同的动作
- **示例**：
  ```
  规则A：如果金额>1000，则打标签
  规则B：如果金额>1000 AND 客户类型=VIP，则打标签
  ```
- **解决**：保留一条规则，或合并规则

#### 3. 包含冲突
- **定义**：一条规则的条件是另一条规则条件的子集
- **示例**：
  ```
  规则A：如果客户类型=VIP，则免运费
  规则B：如果客户类型=VIP AND 订单金额>500，则免运费
  ```
- **解决**：调整规则条件，确保互斥

### 冲突解决策略

#### 策略 1：优先级优先
```
执行流程:
1. 收集所有触发的规则
2. 按优先级排序
3. 从高到低依次执行
4. 遇到冲突时，高优先级规则覆盖低优先级
```

#### 策略 2：最先匹配
```
执行流程:
1. 按规则定义顺序评估
2. 第一个满足条件的规则生效
3. 后续规则跳过
```

#### 策略 3：全部执行
```
执行流程:
1. 收集所有触发的规则
2. 无冲突的规则全部执行
3. 有冲突的规则按优先级选择
```

## 规则依赖关系

### 1. 前置依赖（depends-on）
- **语义**：规则A的执行依赖于规则B的结果
- **方向**：A → B
- **示例**：
  ```
  优惠计算规则 depends-on 资格校验规则
  积分发放规则 depends-on 支付成功规则
  ```

### 2. 触发链（triggers）
- **语义**：规则A执行后触发规则B的评估
- **方向**：A → B
- **示例**：
  ```
  库存检查规则 triggers 库存预警规则
  大额订单规则 triggers 审批通知规则
  ```

### 3. 互斥组（mutually-exclusive）
- **语义**：同一组内的规则只能有一条生效
- **示例**：
  ```
  互斥组: 折扣规则组
  - 会员折扣规则
  - 促销折扣规则
  - 满减折扣规则
  ```

### 规则依赖图

```
库存检查规则
    ↓ (triggers)
库存预警规则
    ↓ (triggers)
补货通知规则

订单金额规则
    ↓ (depends-on)
折扣计算规则
    ↓ (depends-on)
税费计算规则
```

## 规则建模最佳实践

### 1. 规则原子性
- 每条规则只做一件事
- 避免复杂的组合逻辑
- 复杂逻辑拆分为多条规则

### 2. 规则可测试性
- 规则条件应可独立测试
- 规则动作的结果应可验证
- 提供规则的测试用例

### 3. 规则可追溯性
- 规则应有明确的业务来源
- 规则变更应有版本控制
- 规则执行应有日志记录

### 4. 规则文档化
- 每条规则都有详细说明
- 规则之间的关系清晰
- 规则优先级明确
