## 行为建模概述

### 行为的定义
行为是领域对象能够执行的操作或动作，是对对象动态能力的描述。行为建模关注"对象能做什么"而非"对象是什么"。

### 行为的核心要素

1. **行为名称**
   - 定义：行为的唯一标识
   - 规范：动词+名词，如"创建订单"、"确认收款"
   - 特征：具有明确的业务含义

2. **输入参数**
   - 定义：行为执行所需的输入数据
   - 类型：可以是基本类型或对象引用
   - 约束：必填/可选、数量限制

3. **输出结果**
   - 定义：行为执行后返回的数据
   - 类型：成功结果/失败异常
   - 约束：结果格式、有效性验证

4. **前置条件**
   - 定义：执行前必须满足的条件
   - 类型：状态条件、权限条件、数据条件
   - 示例："订单状态必须为'待支付'"

5. **后置条件**
   - 定义：执行后必须满足的条件
   - 类型：状态变更、数据变更、副作用
   - 示例："订单状态变更为'已支付'"

6. **业务逻辑**
   - 定义：行为的具体执行步骤
   - 内容：数据处理、规则校验、关联操作

## 行为识别规则

### 规则 1：动词识别法
从文本中提取表示动作的动词或动词短语：
- **创建类**：创建、生成、新建、添加、插入
- **查询类**：查询、获取、查找、搜索、检索
- **更新类**：更新、修改、编辑、变更、调整
- **删除类**：删除、移除、作废、取消
- **执行类**：执行、处理、计算、验证、确认

### 规则 2：操作对象法
识别"谁对什么做了什么"的结构：
```
主体（对象） + 动作（行为） + 客体（参数）
示例：用户（主体） 提交（动作） 订单（客体）
```

### 规则 3：业务场景法
从业务流程描述中提取关键操作：
- 流程步骤 = 行为序列
- 每个步骤对应一个或多个行为
- 示例："用户填写信息 → 系统验证信息 → 生成订单"

### 规则 4：系统交互法
识别人与系统、系统与系统的交互点：
- 用户输入 → 触发行为
- 系统响应 → 行为结果
- 外部调用 → 行为接口

## 行为签名定义

### 签名结构

```
行为名称(参数列表) -> 返回类型
  前置条件: [条件列表]
  后置条件: [条件列表]
  异常: [异常列表]
```

### 参数定义规范

#### 参数结构
```
参数名称: 数据类型 [约束条件] - 描述
```

#### 示例
```
创建订单(
  客户ID: String [必填, 长度20] - 客户唯一标识
  商品列表: List<商品项> [至少1项] - 购买的商品
  优惠券码: String [可选] - 使用的优惠券
  收货地址: 地址对象 [必填] - 配送地址
) -> 订单对象
  前置条件:
    - 客户存在且状态正常
    - 所有商品有库存
    - 优惠券有效（如提供）
  后置条件:
    - 订单状态为"待支付"
    - 库存预占
    - 优惠券标记为已使用（如提供）
  异常:
    - 客户不存在异常
    - 库存不足异常
    - 优惠券无效异常
```

### 返回值定义

#### 成功返回
```
返回类型: 具体数据类型
返回内容: 详细描述
```

#### 失败异常
```
异常类型: 异常类名称
异常条件: 触发条件
异常信息: 返回信息
```

## 行为条件建模

### 前置条件类型

#### 1. 状态条件
- **定义**：对象必须处于特定状态
- **示例**：
  ```
  支付行为前置条件：订单状态 = "待支付"
  发货行为前置条件：订单状态 = "已支付" AND 库存充足
  ```

#### 2. 权限条件
- **定义**：执行者必须具有特定权限
- **示例**：
  ```
  审批行为前置条件：用户角色 = "经理" OR 用户角色 = "主管"
  删除行为前置条件：用户 = 数据创建者 OR 用户角色 = "管理员"
  ```

#### 3. 数据条件
- **定义**：输入数据必须满足特定条件
- **示例**：
  ```
  转账行为前置条件：账户余额 >= 转账金额
  注册行为前置条件：用户名不存在 AND 邮箱格式正确
  ```

### 后置条件类型

#### 1. 状态变更
- **定义**：执行后对象状态发生改变
- **示例**：
  ```
  支付行为后置条件：订单状态从"待支付"变为"已支付"
  审批行为后置条件：申请状态从"待审批"变为"已批准"
  ```

#### 2. 数据变更
- **定义**：执行后数据发生变更
- **示例**：
  ```
  扣款行为后置条件：账户余额 = 原余额 - 扣款金额
  积分行为后置条件：用户积分 = 原积分 + 新增积分
  ```

#### 3. 副作用
- **定义**：执行后产生的额外效果
- **示例**：
  ```
  订单创建后置条件：发送订单确认邮件
  支付成功后置条件：触发库存扣减行为
  ```

## 行为关系建模

### 1. 调用关系（invokes）
- **语义**：行为A执行过程中调用行为B
- **方向**：A → B
- **示例**：
  ```
  下单行为 invokes 库存检查行为
  支付行为 invokes 发送通知行为
  ```

### 2. 触发关系（triggers）
- **语义**：行为A执行成功后触发行为B
- **方向**：A → B
- **特性**：通常是异步的
- **示例**：
  ```
  订单创建 triggers 库存预占
  支付成功 triggers 发送确认邮件
  ```

### 3. 前置依赖（requires）
- **语义**：执行行为A必须先执行行为B
- **方向**：A ← B
- **示例**：
  ```
  发货行为 requires 支付行为
  退款行为 requires 订单取消
  ```

### 4. 互斥关系（excludes）
- **语义**：行为A和行为B不能同时执行
- **示例**：
  ```
  取消订单 excludes 确认收货
  作废发票 excludes 冲红发票
  ```

### 5. 包含关系（includes）
- **语义**：行为A包含行为B作为子步骤
- **方向**：A → B
- **示例**：
  ```
  下单行为 includes 商品校验
  下单行为 includes 价格计算
  ```

## 行为建模最佳实践

### 1. 粒度控制
- **粗粒度**：完整的业务操作（如"创建订单"）
- **细粒度**：原子操作（如"验证库存"、"计算价格"）
- **平衡**：根据建模目的选择合适的粒度

### 2. 命名规范
- **动词+名词**：如"创建订单"而非"订单创建"
- **业务术语**：使用领域内的标准术语
- **避免技术术语**：如"保存到数据库"改为"持久化"

### 3. 完整性检查
- 每个行为都有明确的输入和输出
- 每个行为都有前置条件和后置条件
- 行为之间的关系清晰明确

### 4. 可追溯性
- 行为可以追溯回业务需求
- 行为与对象、规则、流程的关联明确
